# Feature: Dynamic LLM-Generated Quick Actions

**Status:** Planning
**Date:** 2026-03-24

## Problem

Quick actions currently have two issues:

1. **They show too early.** After the user selects a phase card and sees the greeting, the quick actions immediately appear — before the user has sent a message or received a real AI response. This feels premature and clutters the initial experience.

2. **They're static.** The same 3 buttons appear for the entire conversation based on the initial buyer context. They don't adapt as the conversation evolves. "Compare Prices" is useful in the first 2 messages of a research session but irrelevant 10 messages in when the user is now negotiating.

## Goal

- Hide quick actions until after the user sends their first message and receives an AI reply
- Make quick actions dynamically generated by the LLM based on conversation context
- Order by relevance (most useful action first) — all buttons get the same visual style

---

## Proposed Design

### When to Show

Quick actions should only appear after the first real AI response (not the hardcoded greeting). The flow:

1. User selects context card → greeting appears → **no quick actions yet**
2. User sends first message → AI responds → **quick actions appear for the first time**
3. Quick actions update with every subsequent AI response

### How They're Generated

The LLM generates quick actions via a new tool call, similar to how it already calls `update_deal_phase` or `update_scorecard`. After each response, Claude can call `update_quick_actions` with a list of suggested prompts.

This means:
- Actions are contextually relevant to what just happened in the conversation
- They evolve as the conversation progresses
- Claude decides what's most useful right now, not a static config

### Ordering, Not Visual Priority

All quick action buttons get the same visual style (secondary). Claude orders them by relevance — the most useful next step comes first. Users scan left-to-right; position is the priority signal, not color or size.

Why no visual priority:
- The buyer already knows when they're under pressure — they don't need a red button to tell them
- Priority styling adds cognitive load ("which button should I press?") instead of reducing it
- Quick actions should feel like auto-complete suggestions, not a decision dashboard
- The chat input is the primary interface; quick actions are convenient shortcuts

---

## Answered Questions

1. **How many quick actions should Claude generate?** 2-3. Frontend truncates to 4 and handles 0-1 gracefully.
2. **Should Claude always generate them?** No. Claude should call `update_quick_actions` when the suggestions should change — not on every message. If the buyer asks a factual question and Claude answers, the existing suggestions are probably still relevant. But if the conversation context shifts (buyer arrives at dealership, new numbers on the table), Claude should refresh them.
3. **What happens to the static ACTIONS_BY_CONTEXT config?** Fallback only. Shown on the first real AI response if Claude doesn't call `update_quick_actions`. Replaced permanently once Claude generates dynamic ones.
4. **Should the quick actions persist between messages?** Yes — stay visible until Claude updates them with new ones. But with a staleness rule: if Claude hasn't updated actions in 3+ messages, hide them rather than showing increasingly irrelevant suggestions.

## Technical Approach

### Backend

**New tool: `update_quick_actions`** in `claude.py`:

```python
{
    "name": "update_quick_actions",
    "description": (
        "Suggest 2-3 quick action buttons the buyer might want to tap next. "
        "Call this when the conversation context shifts or the previous suggestions "
        "are no longer relevant — not necessarily after every response. "
        "Order by relevance — most useful action first."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "actions": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "label": {
                            "type": "string",
                            "description": "Button text, 2-5 words. Be specific and actionable, not generic.",
                            "maxLength": 30
                        },
                        "prompt": {
                            "type": "string",
                            "description": "The full prompt sent when tapped",
                            "maxLength": 200
                        }
                    },
                    "required": ["label", "prompt"]
                },
                "minItems": 2,
                "maxItems": 3
            }
        },
        "required": ["actions"]
    }
}
```

**System prompt instruction** — tell Claude to call this tool:

```
- Call update_quick_actions when the conversation context shifts or the previous
  suggestions are no longer relevant. You do NOT need to call it after every
  response — only when the suggestions should change. Order by relevance — most
  useful action first. Labels must be specific and actionable (not generic like
  "Tell me more" or "What else?"). Think of them as button text, not sentences.
```

**No persistence needed** — quick actions are ephemeral UI state, not deal state. They don't need a database column. They flow through the existing SSE tool_result mechanism.

**Schema guardrails:**
- `maxLength: 30` on label, `maxLength: 200` on prompt — prevents Claude from generating essays
- `minItems: 2`, `maxItems: 4` — keeps the action count bounded
- Frontend also enforces: truncate to 4, skip actions with empty label/prompt

### Frontend

**New type in `types.ts`:**
```typescript
interface QuickAction {
  label: string
  prompt: string
}
```

**Chat store (not deal store)** — add `quickActions: QuickAction[]` and `quickActionsMessageIndex: number` state. Quick actions are ephemeral conversation-level UI suggestions, not persistent deal data. They belong alongside messages and sending state in `chatStore`, not alongside deal numbers and scorecard in `dealStore`.

**QuickActions component:**
- Receives `quickActions` array instead of looking up from `ACTIONS_BY_CONTEXT`
- All actions rendered with the same secondary button style
- Order preserved from Claude (most relevant first)
- On tap: sends the `prompt` as a user message (same as current `handleQuickAction`)
- Disabled while AI is streaming (`isSending` guard) to prevent firing messages mid-response

**Transition animations:**
- Quick actions area uses a fade + height animation when appearing, disappearing, or updating
- Prevents jarring layout shifts, especially on mobile where the input jumps up/down
- Consistent with the existing micro-interactions pattern (fade-in on WelcomePrompts, slide-in on session cards)

**Visibility logic in chat.tsx:**
- `showQuickActions` = `quickActions.length > 0 && messages.length > 1 && !isStale`
- No quick actions on welcome state or after only the greeting
- Only appear after a real back-and-forth (user message + AI response)

**Staleness rule:**
- Track `quickActionsMessageIndex` (the assistant message count when last updated) in `chatStore`
- If Claude hasn't called `update_quick_actions` in 3+ AI responses, hide quick actions
- This prevents showing increasingly irrelevant static or outdated suggestions deep into a conversation
- When Claude does call the tool again, actions reappear immediately

**Static fallback:**
- `ACTIONS_BY_CONTEXT` remains as the initial fallback
- On first real AI response (if Claude doesn't call `update_quick_actions`), show the static defaults
- Once Claude calls `update_quick_actions`, replace with dynamic ones
- Staleness rule applies to static fallback too — if 3+ AI responses pass without an update, hide them

**Frontend guardrails:**
- Truncate to 3 actions max (even if Claude sends more)
- Skip actions with empty label or prompt
- Handle 0 actions gracefully (just hide the quick actions area)

### Reliability Considerations

- **Claude may forget to call the tool** in long conversations (15+ messages). The staleness rule handles this gracefully — actions disappear rather than becoming stale.
- **Claude may generate generic suggestions** ("Tell me more", "What else?"). The system prompt explicitly instructs against this. Monitor and tune the prompt instruction as needed.
- **Claude may not call the tool at all** on some responses. This is fine — the tool is optional, and the static fallback covers the first few messages.
- **Token cost**: each `update_quick_actions` call adds ~50-100 output tokens. Since it's not called every message, the cost is bounded. Much cheaper than generating a full greeting via LLM.

### What We're Removing

- The `ACTIONS_BY_CONTEXT` config in `QuickActions.tsx` becomes the fallback, not the primary source
- The static `prompts` Record in `useChat.ts` is no longer needed — the prompt comes from the action itself
- `handleQuickAction(actionId)` becomes `handleQuickAction(prompt)` — no ID lookup needed

### Verification

1. Select a phase card → greeting appears → **no quick actions visible**
2. Send first message → AI responds → quick actions fade in (either dynamic from Claude or static fallback)
3. Continue conversation → quick actions update when Claude calls the tool (smooth transition)
4. If Claude stops calling the tool, actions fade out after 3 AI responses (staleness)
5. All actions have uniform styling, ordered by relevance
6. Tapping a quick action sends the associated prompt
7. Quick actions are disabled/untappable while AI is streaming
8. Labels are short and specific (not generic)
9. Frontend handles edge cases: 0 actions, empty labels, >3 actions truncated
10. Upload a deal sheet photo → AI analyzes → quick actions are relevant to the analysis (e.g., "Is this price fair?", "What fees are hidden?")
11. `make check-all` passes
12. Test with 10+ message conversations to verify Claude keeps generating useful suggestions

## Future Considerations

**Analytics (not blocking v1, but capture what to measure):**
- Quick action tap rate (taps / impressions)
- Which action labels get tapped most frequently
- Dynamic vs. static action tap rate comparison
- Messages before first quick action tap
- How often staleness rule kicks in (indicates Claude forgetting the tool)
- Photo upload → quick action tap sequence (high-value flow)

**Tool count trajectory:**
- Currently at 7 tools after this feature. Monitor response latency as tools grow. At 15+ tools, consider grouping or conditional tool inclusion based on conversation state.
