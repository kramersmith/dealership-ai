# ADR-0007: AI-Generated Panel Cards

**Status:** Accepted
**Date:** 2026-03
**Deciders:** Kramer Smith

## Context

The buyer-facing insights panel originally used a fixed-widget layout: a hero section at the top, followed by tiered rows of purpose-built widgets (deal score, price breakdown, vehicle info, red flags, etc.) arranged in predefined slots. Each widget had its own data-fetching logic and was always rendered in the same position regardless of the conversation's current focus.

This created several problems:

1. **Rigid structure did not match conversation flow.** A buyer discussing financing saw the same vehicle-specs widget in the same prominent slot, even when financing details were the relevant insight. The panel could not adapt to what mattered at each moment.
2. **Adding new insight types required frontend code changes.** Every new widget meant a new component, a new slot in the layout, and layout rebalancing across all existing widgets.
3. **The panel could not reflect the AI advisor's reasoning.** The chat assistant understood the buyer's situation deeply, but the panel was a static dashboard disconnected from that understanding. Warnings, next steps, and tactical advice had no place in the fixed layout.
4. **Phase-dependent composition was impossible.** A buyer researching at home needs a thorough 5-6 card panel; a buyer at the dealership mid-negotiation needs 3 tight cards with scripts and numbers. The fixed layout could not vary its composition by phase.

## Decision

Replace the fixed-widget panel with a dynamic array of typed cards generated by Claude after each chat turn. The system works as follows:

### Backend: separate panel generation call

After `stream_chat_loop()` completes a turn (the main chat response), a separate Claude API call generates the panel cards via `generate_ai_panel_cards_with_usage()` in `panel.py`. This call receives:

- The current deal state (serialized as JSON)
- A truncated window of recent conversation messages (last 2 messages, capped at 300/500 chars)
- The assistant's latest response text

Claude returns a JSON array of card objects. Each card has four fields:
- `type` — one of 8 values defined in the `AiCardType` enum: `briefing`, `numbers`, `vehicle`, `warning`, `tip`, `checklist`, `success`, `comparison`
- `title` — short label (2-5 words)
- `content` — a type-specific content object (schema varies per card type, documented in the prompt)
- `priority` — `critical`, `high`, `normal`, or `low` (controls visual emphasis)

The backend validates each card against the enum values, discards malformed entries, defaults missing priorities to `normal`, and emits the validated array as part of the `tool_result` SSE event. Card usage (tokens, cost) is aggregated into the turn's usage summary.

The panel prompt (~2,500 tokens) uses Anthropic's prompt caching (`cache_control: ephemeral`) so repeated calls within a session pay only for the variable portion (deal state + conversation context). The call uses the primary model (`claude-sonnet-4-6`) and includes `max_tokens` retry escalation for truncated responses.

### Frontend: card type registry

The `AiCard` component (`components/insights-panel/AiCard.tsx`) acts as a renderer that dispatches to type-specific card components via a `switch` on `card.type`:

| Card type     | Component            | Purpose                                          |
|---------------|----------------------|--------------------------------------------------|
| `briefing`    | `BriefingCard`       | Status updates, assessments, next steps, scripts |
| `numbers`     | `NumbersCard`        | Financial data with labeled rows and highlights  |
| `vehicle`     | `AiVehicleCard`      | Vehicle specs, VIN, risk flags, intelligence     |
| `warning`     | `WarningCard`        | Red/yellow alerts for dealer tactics or risks    |
| `tip`         | `TipCard`            | Tactical advice with lightbulb icon              |
| `checklist`   | `AiChecklistCard`    | Read-only action items with progress bar         |
| `success`     | `SuccessCard`        | Celebration cards for wins and savings            |
| `comparison`  | `AiComparisonCard`   | Side-by-side deal comparison                     |

The `InsightsPanel` renders the card array with staggered entrance animations (slide-in with 80ms stagger per card) and exit animations when the card set changes. Each card includes a reply button that opens a `CardReplyInput` drawer, allowing the buyer to ask follow-up questions with quoted card context sent to the chat.

### Prompt-driven composition

The panel prompt encodes detailed rules about card composition by deal phase (research vs. at-dealership vs. financing vs. closing), card stability expectations (data cards like `numbers` and `vehicle` maintain structure across turns; situational cards like `briefing` and `warning` regenerate), priority semantics (critical renders red only on warning cards), and negotiation context integration (scripts, pending actions, leverage points from the deal state drive specific card types).

## Alternatives Considered

### Option A: Keep the fixed-widget layout
- Pros: Predictable rendering, no per-turn API cost, no prompt engineering needed, easier to test
- Cons: Cannot adapt to conversation context, every new insight type requires frontend changes, panel becomes stale or irrelevant as the conversation progresses, no way to surface phase-appropriate composition

### Option B: Fully freeform markdown panel
- Pros: Maximum flexibility for the AI, simplest frontend (single markdown renderer), no schema to maintain
- Cons: No structured data rendering (financial tables, vehicle specs, checklists need visual treatment that markdown cannot provide), no type safety, inconsistent visual quality across turns, no card-level reply interaction, no way to apply type-specific styling (red borders on warnings, green on success)

### Option C: Hybrid — fixed slots with AI-populated content
- Pros: Retains visual consistency of fixed layout, AI fills in content rather than choosing structure
- Cons: Still cannot vary composition by phase (always shows the same slots), wastes space on irrelevant widgets, does not solve the core problem of context-dependent panel composition

## Consequences

- **Positive:** The panel now reflects the AI advisor's understanding of the buyer's situation. Warnings appear when there are genuine concerns, scripts surface during active negotiation, and the card count adapts to the buyer's phase.
- **Positive:** Adding a new card type requires only a new `AiCardType` enum value, a new frontend component, a `case` in the `AiCard` switch, and an addition to the panel prompt. No layout restructuring.
- **Positive:** The card reply system creates a tight feedback loop — buyers can ask about any specific insight card, and the quoted context helps Claude give a focused answer.
- **Positive:** Prompt caching makes repeated panel generation within a session cost-efficient (only the deal state and conversation context are uncached).
- **Negative:** Every chat turn incurs an additional Claude API call for panel generation. This adds latency (~1-2s) and cost. Mitigated by prompt caching and the bounded `max_tokens` (2048).
- **Negative:** Panel quality depends on prompt engineering. Poorly structured prompts or ambiguous card type boundaries lead to suboptimal card selection. The detailed prompt (~240 lines) requires maintenance as card types evolve.
- **Negative:** Cards are fully regenerated each turn rather than incrementally updated. This means the panel flashes/animates on every exchange. Card stability rules in the prompt partially mitigate this by encouraging consistent structure for data cards.
- **Neutral:** The 8 card types and their schemas form a contract between backend prompt and frontend components. Changes to a card's content schema require coordinated updates to both the prompt and the corresponding React component.
